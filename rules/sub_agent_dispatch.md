## Sub-agent调度规则

**委派原则：本体攥住主线——意图、判断、上下文都在本体手里。subagent 领的是「工作包」，不是「思考」。**

委派看两件事：**重量**（会产生你之后不回看的大量输出/探索）和**独立性**（能跟别的活并行）。够重或够独立才派；小活、跟当前对话紧耦合的活、需要全局语境的判断，本体直接做。

能并行的独立工作包，在同一条消息里一起派，别串行排队。

### 各类工作归口的 agent（决定委派后，按此路由）

> 归口 ≠ 必派。这张表只说"如果决定派出去,归谁"。是否派要看下方各自的「何时派」章节,size 不达阈值的都走本体直改。

| 场景 | 归口 |
|------|----------------|
| 商业判断、功能建议、体验改进（产品策略类） | 本体加载 product-strategy skill 主线执行；查证项>2个打包派 general-purpose 去搜 |
| 产品结构梳理、单功能拆解、补AC | 本体加载 product-breakdown skill 主线共创、每步确认每步落盘 |
| 一口气拆解（输入已明确、要隔离重输出或并行批量，如一次补多个功能的AC） | general-purpose subagent，prompt给 product-breakdown skill 路径+输入文件+一口气模式声明 |
| 界面设计 | ui-designer |
| 写代码/改代码 | code-writer |
| 代码审查（工程视角：正确性/AC/设计系统/安全） | code-reviewer |
| 体验审查（用户视角：印象/反馈/状态/文案/流程/认知/情绪/控制感） | ux-reviewer |

本体负责：理解用户意图、做第一手判断、决定走哪条流程、给 subagent 写**带上已探明上下文**的工作包、在节点向用户确认决策点，以及**直接处理够轻的活**（小改、自检、紧耦合的快速判断）。

**违反判断标准（双向）**：

- **漏派**：你在主线里扛了一件会刷屏、或本可以跟别的活并行的重活 → 该派没派，停下来派出去。
- **过派**：你把一个单文件小改、或一个需要全局语境的快速判断，硬塞给 subagent，白付一次启动和交接成本 → 该留没留，拉回主线自己做。

两边都要避免。模糊时，按「这活会不会产生我不再回看的上下文、值不值得那道交接损耗」来判，而不是按它属于哪个品类。

**强交互的活永远不派**：subagent中途没有和用户对话的通道。凡是需要用户逐步确认才能往下走的活（共创式需求拆解、逐方法讨论的商业探讨、需要边看边调的迭代式实现），留在主线：拆解和商业探讨由本体加载对应skill（product-breakdown / product-strategy）跟用户走，迭代式实现由本体直接改；subagent只领「输入已明确、一口气跑完、假设集中列待确认项」的工作包。

**派 ui-designer 或一口气拆解时，本体在派工 prompt 里直接给出当天日期（YYYYMMDD格式）**，不让agent自己取系统日期（跨平台不可靠）。

### mattpocock 工程流路由（编码轨内）

- **触发方式实情**：`/grill-with-docs`、`/wayfinder`、`/setup-matt-pocock-skills` 只能手动斜杠触发；grilling、tdd、research、prototype、domain-modeling 带自动触发（含中文触发词）。撞车裁决：需求拆解归口不变（产品侧 product-breakdown / feature-breakdown），grilling 只接工程设计的把关拷问，不接产品结构梳理
- **工程需求拷问**：动手前用 `/grill-with-docs`（连环追问，落 ADR 和 CONTEXT.md）。产品侧仍走 product-breakdown，两者不混
- **大工程**（≥ 1000 行 / ≥ 20 文件，或超过一个会话装不下）：规划用 `/wayfinder`（工单地图，一张张解决）。首次使用前在目标项目跑一次 `/setup-matt-pocock-skills` 选 tracker，个人沙箱项目选本地 markdown。**大工程的编码必须完整遵循 tdd 红绿循环，无体量豁免**（前提是改动命中「写测试的场景」；纯 UI 样式、文案、配置这类按性质不写测试的场景，不因体量升级为要写测试，判据见下方「何时写测试」）
- **规格等价**：grill-with-docs 拷问产出的结论文档与 ADR、wayfinder 的工单，与 AC 文档（docs/pm-*-ac.md）互为等价规格。注意 CONTEXT.md 是术语表不是规格（domain-modeling 明文规定），所以拷问会话收尾必须把结论落成一份文档（目标、边界、验收要点，落 docs/ 或写进工单），派 code-writer 时给这份落盘物的路径，不必为已拷问清楚的工程需求再走一遍 product-breakdown
- **调研分工**：查仓库、查文档、查 API 事实归 research skill（后台 agent，产出带引用的 repo 内 markdown）。不再使用 deep-research（2026-07-13 用户决定，调研统一走 research）

### 产品类工作的执行方式

本体上下文最全，第一手判断由本体来给，够重才进skill流程。

**本体直接给（不进skill流程）**：

- 快速方向性意见、单个功能要不要加、零散的体验改进点
- 边界清楚的轻量拆解（一两个子功能）

**商业判断 → 本体加载 product-strategy skill 主线执行**：

- 真正的方向性决策（要不要做一个产品 / 一个大功能）、系统性商业论证、用户明确要战略评估
- 每套方法执行完和用户确认再进下一套，确认发生在对话里，零交接成本
- 查证项>2个时按skill里的规则打包派 general-purpose subagent 去搜，搜索刷屏隔离在外，结论带回主线

**需求拆解 → 本体加载 product-breakdown skill 主线共创**：

- 从0出产品骨架（阶段A，方法论 ooux-product-design）或细化功能拆解+AC（阶段B，方法论 feature-breakdown），每步和用户确认，确认一步落盘一步，中断不丢进度
- 产出落盘 `docs/pm-<YYYYMMDD>-<slug>-structure.md` / `-ac.md`

**一口气外派（general-purpose subagent + skill路径）**：

- 输入已明确（用户确认过的场景描述 / 已有structure文件）且产出重适合隔离，或多个独立拆解可并行（如structure确认后批量补5个功能的AC，同一条消息并行派5个）
- prompt里给：product-breakdown skill 路径 + 输入文件路径 + 当天日期 + 一句「按一口气模式执行」
- 它不中途找用户确认，拿不准的集中列在产出文档末尾「待确认项」；本体拿清单找用户批量确认，有修正意见带着文件路径重派，增量修正不从头跑

### 何时派 ui-designer / 何时不派

派 ui-designer：

- 新增完整新页面，需要排版/布局/组件规格方案
- 重做整体视觉风格（换主题、改 design system、统一规范）
- 用户明确要求出设计方案

不派（本体直接做或交给 code-writer 自己处理）：

- 已有页面的样式微调（改色值/间距/字号/圆角）
- 加单个组件到已有页面（沿用既有视觉风格）
- 改文案、改 placeholder、改空状态文字
- 改交互细节（hover/active/loading 状态）
- 任何 UI 调整 < 500 行 且 不改整体布局结构

不派时怎么处理：

- 边界消歧（2026-07-13）：新增**完整**营销页（落地页/官网/招募页整页）默认走上面的「派」，由 ui-designer 按A线 taste-skill 出方案再交 writer 实现；下面这条「本体直接加载 taste-skill」只适用于 <500 行的小体量界面活,或用户明说不要方案文档直接做
- 本体直改：按 code-writer 文件里「前端审美来源」规则,无方案+小调整→沿用项目现状,无方案+新组件→按页面类型选审美skill：营销类页面(落地页/官网/招募页)用 taste-skill 且中文页面配 rules/cn_typography.md 字体补丁,产品UI/看板用 frontend-design(taste-skill 自我声明不覆盖产品UI),细则见 code-writer.md 第7条
- 派 writer：工作包里按同一规则给对应审美skill路径,中文页面附 rules/cn_typography.md,让 writer 自己加载

**Why**：ui-designer 出方案要扫项目设计系统、写完整方案文档、落盘 docs/。对小改动是过度交付,本体或 writer 直接照项目现状改更快。

### 何时派 code-writer / 何时本体直接改

默认：走完整流程做出来的功能（有 AC），由 code-writer 实现。此外满足下面任一条件 → 派 code-writer：

- 改动文件数 ≥ 10 个
- 单文件新增/改动 ≥ 500 行
- 本体没读过且预计要 Read ≥ 10 个陌生文件才能动手（探索成本本身就该隔离出去）

**不构成派工理由**（即使表面上看像，也走本体直改）：

- "完整功能" 但 < 500 行 / 单文件 → 本体直改
- 跨前后端但每端改动都很小 → 本体直改（"多模块交互" 不是单独的派工理由，看体量）
- 本体在前面对话里已经 Read 过涉及的文件 → 本体直改（派出去等于让 writer 重读一遍）

本体直接改（不派 code-writer）：

- 单文件 < 500 行的改动
- 纯 bug 修复且改动 < 500 行
- 纯文案改写、样式微调、配色调整
- 单纯重命名、注释清理、格式整理
- 试验性脚本、一次性任务
- 需要用户边看边调的迭代式实现（如界面效果调优、每轮要用户看效果给反馈），**无论体量**。这是「强交互的活永远不派」在实现环节的应用，派writer等于黑盒等待

本体直接改完，同样按下方 reviewer 触发条件判断是否需要审查，并在交付时做轻量自检。

**Why**：code-writer 从零起步，会把你刚翻过的文件重读一遍。小改用它，又慢又丢上下文——正是要治的两个病。小而紧耦合的改动留在主线，本体带着完整语境直接落地最快。

**调用 code-writer 的工作包模板**：

```
你是 code-writer，实现 <功能简述>。

## 输入文件（自己 Read，不要重新探索）
- AC文档：<docs/pm-*-ac.md 绝对路径>
- UI方案：<docs/ui-*.md 绝对路径>（如有）
- 设计系统：<路径>（如有）

## 已探明的现场（直接采信，禁止重新全局探索）
- <path:line>：<本体已确认的结论，如「入口函数在这里」「这个组件负责XX」>

## 改动范围
- <要改哪些文件/模块，新增什么>

## 边界
- <不要动什么、不要扩散到哪>
```

本体调度前已探明的位置信息必须写进「已探明的现场」，不传等于让 writer 把你刚翻过的文件重新探索一遍。

### 完整流程顺序

> 这条完整流程**只**用于大型改动（≥ 500 行 / ≥ 10 文件 / 跨前后端且每端都重）。非大型改动（单文件 < 500 行的功能、bug 修复、UI 调整、文案修改、跨前后端的小改）**默认全本体直改**，不走这套流程。**例外**：边看边调的迭代式实现（每轮要用户看效果给反馈），无论体量都本体做。

前4站全在主线，确认发生在对话里，不触发派工往返。

1. 阶段判断（主线）：
   - 方向未定（idea/要不要做/该加什么功能/体验怎么改）→ 本体先给第一手判断；够重才进第2步
   - 方向已定但还没有产品骨架（对象/页面/数据模型没梳理过）→ 第3步
   - 已有产品骨架，要细化单个功能 → 第4步
2. 商业判断（主线）：本体加载 product-strategy skill，逐方法和用户走，查证按skill规则外派。输出商业判断+功能建议+体验改进+待验证假设 → 用户决定推进开发后进入下一步
3. 阶段A（主线共创）：本体加载 product-breakdown skill 进阶段A，每步确认每步落盘 `<项目根>/docs/pm-<YYYYMMDD>-<slug>-structure.md`。只要产品骨架就停在这；要继续做某个具体功能就进第4步
4. 阶段B：默认主线共创（product-breakdown 阶段B），落盘 `<项目根>/docs/pm-<YYYYMMDD>-<slug>-ac.md`；输入已明确或批量并行时一口气外派（见上方「一口气外派」）。AC连同待确认项交用户确认，修正直接改文件
5. ui-designer（如涉界面）：传AC文件路径 + 项目设计系统文件路径 + 当天日期，拿回方案文档路径（落盘在 `<项目根>/docs/ui-<YYYYMMDD>-<slug>.md`）
6. code-writer：按上方工作包模板，传AC路径 + UI方案路径（如有）+ 设计系统路径 + 已探明现场。writer 自己 Read 文件，不重新探索
7. 并行判断审查：
   - 满足 code-reviewer 触发条件 → 派 code-reviewer（判定标准见下方）
   - 满足 ux-reviewer 触发条件 → 派 ux-reviewer（判定标准见下方）
   - 两个都需要 → **并行派出**（同一条消息里发两个 Agent 调用，wall-clock 不串行累加）
   - 派 reviewer 时带上启动命令、端口、本次改动涉及的 URL 路径，让 code-reviewer 和 ux-reviewer 各自起服务跑。code-writer 不再起服务、不再截图，运行时验证归 reviewer
8. 审查处理：
   - 全部 ✅ → 交付给用户
   - 任一 ⚠️ 或 ❌ → 合并两份报告的修改清单 + AC文件路径转给 code-writer 修改
   - 两份报告冲突时（罕见，视角不同一般不冲突）→ 本体识别冲突点，带回用户裁决

### reviewer↔writer 循环上限（硬约束，适用于所有 reviewer）

任意 reviewer（code-reviewer / ux-reviewer）→writer→reviewer 整套循环最多走 **2 轮**（即每类 reviewer 最多被调 2 次）。第 2 轮该类 reviewer 仍未给 ✅ 时：

- 停止该类循环，不再自动调 writer 修这一类问题
- 本体把以下信息汇总给用户：第 2 轮该类 reviewer 报告原文、writer 两次改动的对比说明、卡在哪条检查项、各自的理由
- 由用户裁决：接受 reviewer 意见强制改 / 接受 writer 意见放过 / 改 AC 定义或验收口径 / 放弃这个功能

**Why**：writer 和 reviewer 在某条检查项的解读出现分歧时会反复拉锯，无上限地转包会消耗大量 token 也不收敛。两轮足够暴露分歧点，第 3 轮该让人判。code-reviewer 和 ux-reviewer 各自独立计算 2 轮上限，互不影响。

### 何时需要 code-reviewer

默认不调。本体在 code-writer 返回后（或自己改完后）做轻量自检（关键文件 Read 一遍、grep 用户明确要求的几个点），就交付给用户。

只有满足下面任一条件时，才调用 code-reviewer：

- 改动文件数 ≥ 10 个
- 单文件新增/改动 ≥ 500 行
- 安全敏感（认证、权限、用户输入处理、密钥处理、SQL 拼接）
- 用户明确要求审查（"帮我 review 一下"、"严格检查"等）

不调 reviewer 的典型场景：

- 单文件小到中等改动（< 500 行）
- 纯 bug 修复且改动 < 500 行
- 纯文案改写、样式微调、配色调整
- 试验性脚本、一次性任务、原型 demo
- 单纯重命名、注释清理、格式整理

边界情况默认不调，跳过时在交付时一句话说明"本次未走 reviewer，理由是 XX"。例外：模糊情况沾到安全敏感（认证/权限/用户输入/密钥/SQL）仍然倾向调，这是硬触发项，沾边就该查。

### 何时需要 ux-reviewer

默认按改动性质判断。本体调度时按下面规则。

**派 ux-reviewer**：

- 新增完整新流程或新页面（多页面联动 + 跨页跳转，不只是单页改样式）
- 改动关键路径上的核心交互（登录、注册、下单、结算、首屏、核心功能入口）
- 用户明确要求「用户视角看一下/体验怎么样/有没有体验bug」

**跳过 ux-reviewer**：

- 纯后端（API、数据库、定时任务）
- 纯CLI工具（开发者用，不是终端用户）
- 纯配置改动、纯重命名/格式整理
- 单页 UI 调整（不改流程结构，不在关键路径上）→ 本体自检替代
- 单组件改动（增删字段、改样式、改文案、加状态）→ 本体自检替代
- 纯文案微调 → 本体自检替代
- 内部管理后台（用户明确说「内部用够用就行」时跳过）
- 一次性脚本、原型demo

派 ux-reviewer 时，工作包里带上 AC 文件路径 + 本次改了什么 + 启动命令/端口/URL 路径，由它自己起服务跑。

边界情况默认不派，跳过时在交付时一句话说明「本次未走 ux-reviewer，理由是 XX」。例外：关键路径上的改动模糊时仍倾向派。

### 何时写测试 / 谁写测试

测试方法论权威：`~/.claude/skills/tdd/SKILL.md`（mattpocock 版 TDD）。命中「写测试的场景」的编码任务,默认按它的红绿循环走,不再是「实现完后补测试」。code-writer 不写测试,只管实现。

**写测试的场景**：

- 核心业务计算 / API / 路由 / 状态管理
- 复杂验证器、解析器、格式化器（多分支输入）
- 有明确复现路径的 bug 修复（防回归,先写复现 bug 的失败用例）
- 用户明确要求"加测试"

**不写测试的场景**：

- UI 样式 / 交互体验 / hover/active/loading 状态
- 文案 / 提示词 / 系统 prompt
- 配置文件（models.json、环境变量、theme 配置）
- 重命名 / 注释清理 / 格式整理
- 试验性脚本、一次性任务、原型 demo
- 简单纯函数 / 小工具函数（< 50 行,输入输出明确）

**怎么写（前三条是 tdd skill 的硬规则,第四条是本体系的体量分级;动手前 Read `~/.claude/skills/tdd/SKILL.md`）**：

- **先定 seam 再写测试**:测试只写在公共边界(seam)上。动手前列出要测的 seam 跟用户确认,没确认的 seam 不写测试。不追求全覆盖,测试预算花在关键路径和复杂逻辑上
- **垂直切片**:一条失败测试→最小实现让它变绿→下一条。禁止一次把测试全写完再实现(horizontal slicing,它明令的反模式:批量测试验证的是想象中的行为)
- **重构不进红绿循环**,归审查阶段(code-reviewer agent 或本体自检;tdd skill 原文提到的 code-review skill 在本体系的对应物就是 code-reviewer)。断言禁止 implementation-coupled(mock 内部协作者/测私有方法/绕过接口查数据库)和 tautological(用代码同样的算法算期望值),判据见 `~/.claude/skills/tdd/tests.md` 和 `mocking.md`
- **体量分级**:本体直改阈值内的小修复(单文件 < 500 行),且 seam 就是被修函数或接口本身、无歧义时,seam 自确认不必停等,汇报里说明测了哪个 seam 即可;大工程(≥ 1000 行 / ≥ 20 文件或 wayfinder 级)必须完整走 seam 确认和红绿循环,无豁免

**谁写测试**：

- **默认本体写**：按红绿循环,测试和实现在本体手里交替推进
- **派 code-writer 的改动**：本体先按 AC 定 seam 写失败测试(红),工作包里给测试文件路径,writer 的完成标准是让测试变绿,writer 不许改测试
- **改动非常大（≥ 1000 行 / ≥ 20 文件或 wayfinder 级）**：没有体量豁免。先用 /wayfinder 把大工程拆成工单,每张工单内按红绿循环推进（本体写 seam 失败测试,自己实现或派 writer 让测试变绿）。tdd 的垂直切片本来就把大工程切成小片,「太大所以并行批量写测试」是 horizontal slicing 反模式,禁止。旧的并行派测试 subagent 模板已随本条废除（2026-07-13,理由:批量黑盒测试验证的是想象中的行为,与 tdd 硬规则直接冲突）

### 什么情况下不需要完整流程

| 场景 | 处理方式 |
|------|---------|
| 纯产品idea/商业判断/功能建议/体验思考 | 本体先给第一手判断；够重才进 product-strategy skill，且不进入实现 |
| 纯需求拆解（用户只要拆解结果不进开发） | 边界清楚的轻量拆解本体直接列；够重则本体加载 product-breakdown 主线共创，按需走阶段A或B |
| 用户只要产品骨架（对象/页面/数据模型，不细化具体功能） | product-breakdown 阶段A主线共创，停在数据模型草案 |
| 用户已有产品骨架，只补单功能拆解 | product-breakdown 阶段B主线共创；输入明确或批量并行时一口气外派 general-purpose |
| 纯bug修复（< 500 行 / 单文件） | 本体直接改 + 自检；涉及界面时按条件判断 ux-reviewer |
| 较大 bug 修复（≥ 500 行 / 多文件） | code-writer → 按条件判断 code-reviewer + ux-reviewer |
| 纯样式微调（配色、间距、单文件小改） | 本体直接改 + 自检 |
| 较大样式/界面改动 | ui-designer（如需方案）→ code-writer → 按条件判断 code-reviewer + ux-reviewer |
| 已有界面的体验诊断 | 直接调用 ux-reviewer 或 ux-bug-check skill，不走其他agent |

### 调用code-reviewer的硬约束

历史教训：一次审查17个文件、未告知具体行号时，reviewer陷入死循环重复Read同一文件前30行115次。本体在调度前已掌握全部位置信息，不传过去就是把定位成本压给reviewer。

**1. 默认单批审查**

不要为了"加速"主动拆批。三批并行的 wall-clock 受最慢一批拖累，但 token 消耗是单批的 3 倍，ROI 很差。先按单批发，让一个 reviewer 把全文走完。

只有以下情况才考虑拆批：

- 改动文件数超过 10 个
- 单个文件改动超过 2000 行
- 改动涉及多个互相独立的模块（前端 + 后端 + DB schema 同时改）

拆批时按"独立可审查单元"拆，不要按文件数硬切：

- 单元内的文件能闭环完成审查（前后端一致性、调用方与被调方的核对，必须在同一批里能同时看到）
- 单批文件数不超过 6 个
- 单批审查重点不超过 4 条

例（确实需要拆批的）：5 套模板共 15 个文件（前端 Vue + 前端 CSS + 后端 CSS.ts，每套 3 个），按"每套模板一批"拆成 5 批，每批 3 个文件。

例（不该拆批的）：1 个单文件 HTML 共 1800 行的销售落地页，单批走完即可，不要按 section 拆 3 批。

错误拆法：按"前端 Vue + CSS 全部一批、后端 CSS 全部一批"——这样前后端一致性永远没法在同一次审查里核对。

**2. prompt 里必须给具体行号区间**

对每个修改文件，prompt 中必须明确写出「要查看的行号范围 + 该位置应该是什么样」。新建文件不需要给行号（reviewer 会整文件 Read）。

**3. 调用 reviewer 的 prompt 模板**

```
你是 code-reviewer，审查 <功能简述> 的代码改动。

## 新建文件（请整文件 Read）
- <path1>
- <path2>

## 修改文件（已标注关键位置）
- <path>
  - 第 X-Y 行：<改了什么，应该长什么样>
  - 第 A-B 行：<改了什么，应该长什么样>

## 删除文件
- <path>（确认相关引用已清理）

## 审查重点
1. <重点1>
2. <重点2>

请逐个文件审查，输出 markdown 报告。
```

**4. 写完 prompt 自检**

发出 Agent 调用前问自己：

- 我是不是在没必要的时候拆批了？默认单批，除非触发上面三个拆批条件
- 如果拆批了：这批文件能闭环审查吗？前后端一致性、调用方与被调方都在同批里？
- 单批文件数超过 6 个，或审查重点超过 4 条了吗？超过就拆
- 每个修改文件都标了行号吗？没有就回到代码里查清楚再传
