# UI文案专业度检查

蒸馏自用户真实产物的实测扫描（2026-07-14，课件38页、产品UI36个组件、落地页各一轮），v1.1按留出集A/B验收的裁判证据修订（硬线改复查信号、语域基准修正、检查范围含JS字符串）。实测规律：按钮和徽标通常已经专业，病灶集中在空状态、提示条、确认框、tooltip这四类「系统要开口说话」的场景。

**检查范围**：一切渲染给用户看的文字，包括JS/TS里拼接的字符串（confirm、alert、toast、动态状态文案），不只HTML静态节点，只扫HTML会漏掉一半病灶。aria-label、alt等辅助技术文本不算渲染可见文字，不按本清单报（它们的职责是如实描述，按可访问性口径另评）。

## 核心判断

**界面文字是产品在陈述状态，不是人在跟用户说话。**官方感的来源不是词藻，是三件事：短（字数有硬线）、定（陈述和祈使，不安慰不商量不卖萌）、名词化（标签是名词，按钮是动宾）。

病根只有一个：把「怕用户不懂」翻译成了「多写几句」。正解永远是让控件和结构自己说话，机制说明进文档，不进界面。

## 七类检查

每类给检测特征和重写原则。检测时逐类过，同类问题合并报。

**1. 解释机制上墙**
- 检测：tooltip、空状态、状态条、通知里出现内部机制、副作用、兜底路径的说明（「会同步XX」「AI会先检查XX」「数据从XX里取」）
- 重写：只陈述状态或动作结果。tooltip是名词短语或短祈使句；空状态是一句事实加一个入口

**2. 教学腔**
- 检测：状态文案里塞操作口令（「再点一下停止」）；确认框里教补救技巧（「可用快捷键撤回」）
- 重写：状态归状态，操作方式靠控件自身传达；补救入口做成按钮，不写成课文

**3. 聊天腔与拟人化**
- 检测：界面骨架文字（标题、通知、字段标签）用第一或第二人称发声、带语气副词和安慰语（「悄悄」「别担心」「帮你」）、先安慰后说事
- 重写：中性陈述。对话人设只允许出现在真正的对话气泡里，气泡外的一切文字都是产品在说话

**4. 语域不齐**
- 检测：同组按钮里书面短语混口语单词；placeholder与label语域跳脱（label是正式名词、placeholder是聊天问句）；同一概念在不同界面叫法不同（说明区一个名、按钮一个名、设置页又一个名）；演示卡和模拟截图的语气与真实产品文案不一致（演示在替产品发言，必须同声）
- 重写：语域基准是产品全局的语气定位，不是页内最正式处。只报页内跳脱、跨页术语错位和演示与真品的不一致

**5. 开发者黑话泄漏**
- 检测：括号式标注（可空、选填除外）、版本号、内部状态名（触达中）、调试式占位（缺内容、找不到）
- 重写：换用户语言（选填、发送失败），或该字段直接不展示

**6. 自言自语与辩解**
- 检测：副标题或正文解释编辑取舍过程（「先放5个最能体现…」）；对不存在的信息补说明（「未公开XX」）
- 重写：直接给结论；不展示的东西直接不展示，不解释为什么不展示

**7. 字数与形态线（复查信号，不是自动违规）**
- 信号：按钮>4字、字段标签非名词、空状态正文>1句、确认框超过一句后果加一个问题、标题含「将」「会为您」「即可」这类许诺式措辞
- 触线后先判断：能在不损失必要信息的前提下缩短才报（错误：把「最小化到托盘」报成超字数，六个字全是必要信息；正确：「点击这里提交你的表单」缩成「提交」）。功能性长按钮不是病，许诺式空话才是

## 严重度（对齐ux-bug-check三级）

- **阻断**：文字导致误操作或看不懂当前状态（黑话占位符、状态误导）
- **困惑**：要琢磨才能懂（机制说明淹没关键信息、语域混乱）
- **不爽**：能用但不专业（拟人卖萌、教学腔、辩解）

## 例外（防过度矫正，逐条硬性）

1. **破坏性操作的确认框必须带后果和目标识别信息**（名称、时间），这是决策必需信息不是解释腔。只有一句静态文案的删除确认反而是问题（见ux-bug-check的user-control清单）
2. **真正的对话气泡**（AI作为对话方说的话）归rules/no_ai_style.md管，不按本清单改成机器腔
3. **引导页和onboarding**的教学内容放专门的引导组件里不算违规，但不得混进常驻状态文案
4. **课件上墙字**的口径以live-lesson-deck的teaching-voice.md为准，两者冲突时它优先
5. **营销页hero**允许长句和情绪，但必须是断言不是解释；判营销文案先看taste-skill第4.9节是否已覆盖，不重复报
6. **整页统一的叙事语气与语域**：故事页、引导演示页刻意用口语风格且正文与控件互相呼应的，是设计不是病；整页或整区块统一用外语（如footer整行英文眉题）同理。只报这类页面和区块内部的跳脱

## 输出

每条发现：位置(path:line) / 类别(1-7) / 原文（≤20字截断） / 为什么不专业（一句话） / **重写后的文字**（直接给可替换的成品，这是本skill与ux-bug-check「只给方向」的区别：文案检查的交付物就是新文案）。

重写的文字本身必须过第7类硬线和 `~/.claude/rules/no_ai_style.md`（禁破折号、禁双引号、禁油腻煽情、禁自我标榜、中文与数字英文间不加空格等全部条款）。报告输出同样遵守no_ai_style。

## 边界（不做什么）

- 不管视觉（字号、色值、间距归ui-designer/设计skill）
- 不管工程正确性（归code-reviewer）
- 不管对话内容和长文（归no_ai_style和相应写作skill）
- 不重写信息架构（哪块内容该不该存在归ux-bug-check流程类）
