# 直播课件

目标是交付一份能直接上课的网页课件：浏览器打开、左右翻页、像 PPT 一样播放，并且有讲师能对着讲的课程逻辑。

本 skill 最重要的工作不是排版，而是把项目或主题整理成学员能学会的认知顺序。

## 先分流

收到任务后先判断是哪一类：

- **新建课件**：从项目、主题、文章、案例或经历生成一份新的 `lesson.html`。
- **修改已有 deck**：在已有 `lesson.html` 上换案例、插页、删页、重排、改文案、加编辑器或修交互。

两类任务共用教学判断，但执行重点不同：

- 新建课件先定课程主线，再制作页面。
- 修改已有 deck 先识别页间耦合，再动结构。

## 核心判断

### 主线是一条追问链，不是章节分类

主线不是把内容按主题分类摆在一起，是一条不断被「为什么现在需要它」驱动的因果链：学员站在上一步的终点，自然会冒出下一个问题，下一段内容就是在答这个问题。检验一段内容该不该留在主线上，只有一个问法，它是不是在回答学员刚冒出来的那个问题；如果只是主题相关、顺手讲讲，内容本身没错也是主线上的赘肉，该删或挪到真正需要它的地方。这条链哪一环该出现、先后怎么排，要按这节课的实际内容判断，不是套死一个模板。

产品定位是这条链上最容易被用错的一环。它不是开场讲一次就收工的口号页，是贯穿全程、反复被当尺子拿出来用的锚点，要不要做这个功能、这一页该不该出现、模块该不该砍，都要能被定位解释清楚。定位出现的时机也不必在开场：可以先带学员做出第一个具体的东西，等他冒出「这就够了吗，还要做什么」这个问题时，再回收定位去回答「接下来每一步为什么存在」。这跟备课时先想清楚功能、再倒推出一句站得住的定位是同一个逻辑，只是讲给学员的顺序倒过来，呈现顺序服务「学员好懂」，不必忠于备课时真实的推导顺序。

技术可行性验证也要挂在这条链上，不是单独插进来的一段技术教学。它要紧跟在学员已经对要做的功能有了初步设想之后，回答的是「这些功能能不能做出来」，不是脱离功能设想单独讲一段原理，讲成一节孤立的技术课。

### 教学顺序不是制作顺序

项目怎么做出来，通常不等于学员怎么学会。严禁按工作日志、提交记录或制作时间线铺页。

主线必须是一条有方向的递进，不是一组并列模块。

这是追问链原则最常被打破的方式：拿时间线当主线，页面顺序就变成「发生了什么」，而不是「学员需要先弄懂什么」。

### 定位陈述要经得起追问

课程里要给某个项目定一句定位，这句话得扛住「这算产品还是作品」的追问。只解决讲述者本人或所在团队内部不便的东西，是个人作品/工具，不要借「产品」的措辞把格局撑大；真要叫产品，就得讲清楚服务的是谁的什么痛点，不能停留在讲述者自己的个体困扰上。定位撑不住时，宁可老实定义成个人工具，也不要用产品话术包装一个自用小脚本。

### 新概念要挂在学员的旧认知上

课程要引入学员没接触过的技术名词或机制时，先找一个学员大概率听过、但没细想过的旧概念，把新东西挂上去，让学员的反应是「原来是我知道的那个东西换了个做法」，而不是硬塞一个新名词让他记。同理，「遇到不确定的事，先丢给AI探路」这类可迁移的思维动作，如果学员不会默认想到，要单独用一页把这个动作本身点出来，尤其当它呼应系列课程里已经立过的通用原则时，要点名接上，不能指望学员自己联想到。

### 你也是知识输入源

用户给的是项目事实和个人方法，一门课还需要机制、原理、术语和迁移方法。能把学员从「知道怎么做」带到「知道为什么」的内容，要主动补。补的判据：每个补充的机制点要能独立撑起一页，配图形或类比讲透；检验问法是学员只看这一页能不能答出「为什么」，答不出就还是点缀。只听用户口述指令、原样转成页面而不做知识增值，是明确的失败模式。

补充内容按两类处理：

- **通用机制 / 概念 / 原理**：可以补，但要核权威源；没核实或只是推测，就标明。
- **项目里的坑、数字、故事、产出**：只认用户给的和项目里真实存在的，一律不编。

### 方法和案例要分层

可迁移的方法论先独立讲清，再用真实案例实操。不要把方法和单个案例的演化搅成一页，否则学员会以为方法只适用于这个案例。

如果一份课同时有通用方法和案例细节，视觉上也要让两层可区分，例如固定眉题、标签、步骤轨或版式差异。

走追问链时，通用方法不必在链条中途就摊开讲透，可以先埋在具体案例的应对方式里，到收束阶段再单独提炼成一页方法论，让学员对上「原来刚才那几步都是在用这个方法」。

## 工作流

贯穿全程的确认门槛：deck是重交付物，主线或风格错了就是整份返工，确认一次的成本远低于返工一次。课程主线（第2步）、知识点清单（第3步）、页序大纲（第4步）、草模走查（第4.5步）、风格预览（第5步）是五道必须拿到用户明确认可才能通过的门，AI自己判断「已经够清楚」不算数。只有在没有对话通道（被当作subagent一口气执行）时，才允许用保守假设过门，并把假设集中写进待确认项。

确认内容的载体：短的直接发在对话里；预计超过约30行的清单或大纲落盘到 `docs/lesson-<YYYYMMDD>-<slug>-plan.md`（知识点清单和页序大纲写同一份文件，先清单后大纲，追加不覆盖），对话里只发摘要加文件路径，用户可以直接在文件里改。用户提修改后，只重发被改动的行或板块，不重发全表。

### 1. 收集上下文

主动凑齐这些信息：

- **学员侧**：学员是谁、基础如何、主动来学还是被动刷到。
- **课程定位**：这节课在系列里负责什么、讲什么、不讲什么。
- **学员卡点**：学员最想学到什么，最容易卡在哪里。
- **项目侧**：读 README、开发日志、设计文档、核心代码、测试和成品，找出亮点与可复用方法。
- **外部参照**：同主题教程、产品文档、概念解释，不确定的新信息要查权威来源。
- **你的知识**：把相关机制、概念、术语补成知识点，补之前做必要核验。

缺关键答案就问；没有对话通道时，用保守假设继续，并写进待确认项。

### 2. 定课程主线

主线由三件事推出：

- 这节课的角色和重心
- 学员最想获得的能力
- 学员最大的卡点

主线没定下来，不要排页。这里定的是你自己想清楚的主线，不是页面呈现顺序；定位要不要延后揭晓这类呈现层面的决定，留到第4步排页时再按学员认知定。

主线定完不代表能直接排页：把主线压成一句话（这节课从哪个认知状态讲到哪个认知状态），拿这句话找用户确认，确认了才进入知识点清单和排页。AI 自己判断「主线已经清楚」不算数，只有用户认可才算数。确认后如果主线又要推翻重排，当成新一轮回到本步骤处理，不要只在页面层小修小补掩盖主线问题。

常见主线：所有主线都是追问链，区别只在链条从哪个问题起步，不是三种互斥的模板。

- **主动学习者**：链条从「方法论是什么」起步，先给高密度框架，再用案例往下验证。
- **被动观看者**：链条从「痛点是什么」起步，先抛出具体痛点，再用解法逐层往下展开。

教「从真实卡点出发，用AI做出一个东西」这类课，是本skill最常见的场景，属于被动观看者这条路的一种具体展开，追问链大致长这样：

真实卡点，具体到扛不住的场景 → 现成方案够不够，逼出自己做的必要性 → 自己做要做成什么样，也就是定位，可以先按最小功能设想过一遍再回收 → 做不做得出来，技术可行性挂在已有的功能设想上问，不是独立技术课 → 做出最小的一块，带做，只求跑通 → 这就够了吗，追问引出定位要解释的下一批功能 → 把功能做完整，带做，交付级产品 → 这套方法换个场景还成立吗，迁移

链上每一环存在的理由，是回答上一环自然冒出来的问题，不是「这个话题也值得讲讲」。哪一环要不要出现、要不要合并、先后顺序怎么排，按这节课的实际内容和学员认知水平判断，这只是常见形态，不是固定模板。

### 3. 列知识点清单

列知识点不是把用户输入和项目里已有的知识登记一遍，是用困惑链倒推找出学员真正缺的知识：

1. 从学员这节课要做出的东西出发，拆成几个能力块。例：B站弹幕实时看板，拆成数据获取、数据存储、看板呈现。
2. 对每个能力块追问「学员要做到这件事，还缺什么知识」，一层层往前倒推。例：数据获取缺爬虫，爬虫缺HTTP轮询和WebSocket的区别。推到学员已有认知为止，停点就是「新概念挂在旧认知上」的挂钩位置。
3. 把倒推出来的知识收敛成板块，板块内列具体知识点。

要求：

- 每个知识点都能支撑课程定位。
- 每个知识点都能落到具体页面。
- 缺失的机制、原理、术语主动补齐，每个补充点按「能独立撑起一页」的判据给足体量（见核心判断「你也是知识输入源」）。
- 项目私有事实不补、不编。
- 命名口径：板块名用行业通用词（产品定位、MVP、黑话解析、项目开发、方法总结这类一听就懂的词），不自造口语化标签；具体知识点不起自造方法名（如「需求四问法」），直接用一句大白话说清这条讲的是什么，让用户看一眼就能判断要不要留。

确认形式固定如下，不要自选形式：

1. 先给倒推视图，每个能力块一条链，让用户一眼看到「要做成这件事到底缺哪些知识、知识之间怎么关联」，例：

   ```
   数据获取 ← 缺：爬虫是怎么回事 ← 缺：HTTP轮询和WebSocket的区别
   数据存储 ← 缺：数据库最小知识（建表、写入、查询）
   看板呈现 ← 缺：图表选型 ← 缺：实时刷新机制
   ```

2. 再给分板块表格，每个板块一张表，四列：知识点、学完达到什么状态、挂在主线哪一环、来源。编号跨板块全局连续，方便用户按编号提修改。来源只允许四种标注：项目事实、用户口述、AI补充已核、AI补充未核。
3. 表格后固定跟「待确认项」，集中列AI拿不准的假设。
4. 固定问法收尾：直接回复确认，或按编号说改哪条。

清单必须等用户确认，确认后才进入排页；只有无对话通道时，才按保守版本继续，并集中列待确认项。

### 4. 排教学顺序

排页的核心是接好追问链。链条从哪个问题起步，第2步定主线时已经定死，这里不重新选形态，只管把链接实。

通用规则：

- 排完页顺一遍这条链：每一页是不是在回答上一页冒出的问题；接不上的页面，哪怕主题相关，也要挪位置或删。这一步比选哪种起步形态更重要，起步选错还能救，链条断了就是主线出了问题。
- 每个知识点先讲为什么需要，再讲它是什么。
- 开场建目标感：第一个知识点之前，先让学员看到终点（成品先行）或被一个真问题勾住，没有动机的开场学员不进入状态。
- 检验跟着产出走：学员要亲手做出东西的环节配验收位（带做验收清单或作业）；纯认知内容不设当堂检验，靠追问链回扣，下一环接得上就说明上一环立住了。不逐知识点设检验。
- 一次只引入一个陌生概念，难点之间用低负荷的点做缓冲。
- 一个页面只承担一个清晰认知任务。
- 成品、案例、演示页按主线需要放置，不为好看单独占页。
- 目录按学员心智切成 4 到 6 个章节，不写「钩子」「铺垫」这类讲师术语。
- 一组步骤或部件要有总览页，钻进去的每页用步骤轨回扣全貌。

排完页把页序大纲给用户确认，确认形式固定如下，不要自选形式：

- 开头一行总量：共几页、几章、预计几分钟。
- 按章节分组，每章标页码范围和预计分钟数，章内表格四列：页码、页标题、这页回答的问题、页型与素材需求。「这页回答的问题」写上一页冒出的那个问题，追问链断没断，用户扫一遍就能看出来。
- 表格后列「素材缺口」：哪几页缺真实截图、logo、回放素材。这块是知会不是门槛，默认用占位继续做，占位页交付时集中列入待确认项。
- 固定问法收尾：直接回复确认，或按页码说改哪页。

用户认可后进入草模走查（第4.5步），不直接进制作。

节点图、结构图、方法论图里的小框标题控制在 6 到 7 个汉字以内。超长就加宽节点，或把说明移到节点外。

### 4.5 草模走查（结构冻结门）

页序大纲确认后先出草模deck给用户走查，不直接制作。教学判断只有在「像上课一样走流程」时才真正被激活，文字大纲激活不了它；跳过草模，顺序问题会在成品deck上暴露，一次模块级重排是百页级联动（页序、SECTIONS、承接文案、全页验证），在草模上只是几分钟。

- 草模 = 真实壳 + 骨架页：翻页、目录、进度条、章节全部可用，每页只放页标题、「这页回答的问题」一行、素材类型的灰色占位块，不做视觉、不写正文。壳直接按固定舞台标准搭（见第5步），锁定后同一份壳铺内容，不重建；样式用素坯，不代表最终风格，风格确认仍在第5步。
- 用户像上课一样从头翻到尾走查，检验口径：每翻一页能顺畅说出下一句口播；讲不顺的地方就是链断的地方，当场调顺序。
- 走查阶段鼓励反复翻顺序，这是用户想清楚教学法的过程，不算返工；调整用 `scripts/deck-surgery.mjs` 按 slide id 移页，分钟级。
- 过门标准：用户明确说锁。AI判断「已经顺了」不算数。
- 锁定时把每个定序决策的理由用一句话记进 plan 文档（例：功能先于对象，因为对象是从功能清单里圈出来的）。锁定后再要动模块级顺序按例外处理：先把当时的定序理由翻出来对质，新论点成立才动，防止顺序在两个方案之间来回震荡。
- 锁定后进入第5步制作，按模块铺内容，一个模块铺完给用户看一轮；这时的反馈是页内修改，不再引发级联重排。

### 5. 制作 deck

视觉、舞台、翻页、动效、字体和配色交给 `frontend-slides` skill。

使用方式：

- 第 4 步动手前加载 `frontend-slides`。
- **固定舞台是硬要求，不是可选**：整份 deck 放进一个固定尺寸（如 1920×1080）的舞台，用 transform:scale 等比缩放铺满窗口，不足比例居中留边。舞台内正文字号、间距、留白一律用相对舞台的单位（px，或把舞台设成 container 后用 cqw/cqh），禁止用视口相对单位 vw/vh 和 clamp(...vw...) 做内容尺寸。为什么必须这样：不做固定舞台，布局会随窗口的尺寸和比例实时重排，预览面板、投影、验证脚本各是一个窗口，同一页在这个尺寸挤满或溢出、在另一个尺寸正常，作者按单个尺寸验证根本抓不到别的尺寸的问题。做了固定舞台就所见即所得，任何窗口只是整体缩放，验一个尺寸等于验所有尺寸。具体舞台模板见 `references/fixed-stage.md`。
- 只跳过 `frontend-slides` 的 Phase 0 和 Phase 1（模式判断、内容与密度问卷），内容侧已由本 skill 定完；风格环节（Phase 2）不跳。
- 风格必须经用户确认才能生成整份 deck。用户没指定风格时，按 `frontend-slides` Phase 2 出3版差异化首屏预览让用户挑；用户点名了风格或项目有锁定的视觉基线时，按该风格实作1版首屏预览请用户认可。不要因为「基线已存在」或「用户提过一个方向」就自行判定风格已定、跳过预览，风格成不成立只有用户看过实际画面才算数。

### 5.1 UI体系要按页型落地，不按单页美化

做 deck 时先把页面按教学功能归类，再给每类页型定一套可复用视觉系统。不要只把几个显眼页面单独做漂亮，否则整份课件会变成风格拼贴。常见页型至少包括：

- **封面 / 章节转折页**：负责建立题目和节奏。首屏要有项目或主题的第一视觉信号；章节转折页要让学员知道「现在在问什么、下一段回答什么」。
- **问题页 / Q页**：负责追问链转场。统一成一套可识别的问题系统，例如编号、阶段标签、主问题、下一步提示。主问题可以强，但辅助信息要轻，避免像普通大标题页。
- **截图 / 产品成果页**：负责展示真实产出。统一用截图框或设备框承载，包含轻量标题栏、窗口点、来源或状态标签；不同项目按领域适配外观，例如数据产品像仪表盘窗口，移动产品像手机框，文档产品像页面预览。不要让同一份 deck 里一会儿裸图、一会儿卡片、一会儿浏览器框。
- **技术动画 / 机制解释页**：负责把抽象机制讲成人能看懂的流动过程。统一放进「演示舞台」：浅色或暗色画布、稳定边框、标题栏、统一阴影、统一节点和箭头语言。动画内部内容随主题变化，但舞台容器要一眼看出这是机制演示。机制动画不自动播：接到翻页键节奏上，讲师按「下一步」先把当前动画完整播一遍，再继续揭示或翻页；只有氛围装饰和成品 live 演示允许自动循环。
- **步骤 / 流程页**：负责让学员不迷路。用步骤轨、流程轨或面包屑回扣全貌，轨道存在感要弱于当前页标题和主体。
- **总结 / 作业页**：负责收束和行动。总结页把方法抽象出来，作业页展示交付标准和真实成果示例，不要只放一句口号。

### 5.2 风格适配规则

UI体系要抽象成「结构角色」，不是抽象成某个 deck 的具体颜色或文案。迁移到新课件时按下面顺序适配：

1. 先确定主题气质：是数据看板、工具软件、内容创作、业务分析、生活助手、硬件设备，还是别的类型。
2. 再定一组项目主色和辅助色，主色只用于识别和强调，不要让整份 deck 变成单色主题。
3. 把截图框、问题页、机制舞台、步骤轨都换成同一套主色语言，但保留各自的信息层级。
4. 根据素材类型换外壳：网页截图用浏览器窗，移动截图用手机框，命令行用终端窗，文档或报告用纸张/报告壳，流程机制用演示舞台。
5. 对已有 deck 做视觉升级时，优先统一页型和容器，不先改文案；文案改动会牵动课程主线，除非用户明确要求。

### 5.3 视觉层级收口

一份 deck 最后要做一次视觉收口，至少检查这几类：

- **进度轨弱化**：章节轨、问题轨、面包屑是导航提示，不是主视觉。完成态用低饱和灰或淡色，当前态用细边框或浅底强调即可，避免实心高饱和按钮抢标题。
- **截图框统一**：同类截图用同一框架和同一阴影等级；框内图片不要再额外套不同圆角和阴影。截图框标题栏里的文字短，说明性文字放到旁边或下方。
- **技术动画统一**：所有机制动画页共用同一演示舞台外壳。节点、连线、标签、提示气泡的圆角、边框和阴影要统一；只有内容和颜色强调随知识点变化。
- **问题页成体系**：Q页不要散落成不同版式的大字页。编号、阶段标签、主问题、下一步提示统一；是否显示大号水印编号按风格决定，但同一 deck 内保持一致。
- **首屏非空**：每页主视觉和核心内容立即可见，frag 只给次要 remark 做逐步揭示；新页做完核对首屏不是空的，不要让学员先看一页空白再按出内容。
- **响应式收口**：固定舞台 deck 按舞台设计尺寸验；非固定舞台或旧 deck 改造时，至少用一个标准宽屏和一个窄预览宽度验关键页，尤其是截图页、技术动画页、四列卡片页、总结页。

素材原则：

- 真实产品、配置、代码、文档，尽量展示真实内容，不只写文字描述。
- 讲外部产品时用真 logo。logo 放 `assets/<category>-logo/`，HTML 用相对路径引用。
- 如果确实无法取得素材，用占位，并写入待确认项。
- 需要展示「AI 调试 / 迭代 / 带做实操」这类人和 AI 真实协作的动态过程时，优先做成真实会话回放（提取真实 session 转写，做成可点开的外链页面或卡片），不要包装成提示词卡片＋学员自练这种抽象呈现来代替真实过程；提示词卡片只用于单纯要学员照抄执行、不需要展示协作过程本身的场景。回放页型和同步注意事项见 `references/page-patterns.md`。

载体原则：

- 核心运行时是一份 `lesson.html`，浏览器直接打开即可放映。
- 图片、logo、视频等静态素材可以随同 `assets/` 交付，但不能依赖本地服务或构建步骤。
- web 字体可以外链，但必须带系统字体 fallback，断网也能放。
- 每个 slide 给稳定 id（如 `d1-s8`）：页码会随插删页漂移，文档、沟通、特殊页定位都用 id。
- 不做讲师注记：deck 不内置 `NOTES` 数组和注记抽屉，讲师不用它。解释性、展开性内容不上墙也不落注记，直接不产出；讲法取舍见 `references/teaching-voice.md`。

代码展示原则：

- 可以展示真实代码、配置、文件结构和文档片段。
- 讲解主线不要围绕变量名、函数名、类名展开，要讲行为、结构和原则。

### 6. 内置作者编辑模式

新建课件默认内置作者编辑模式；修改已有 deck 时，如果用户要编辑能力，也按同一模板补齐。

使用本 skill 自带模板：

- 模板目录：`templates/deck-editor/`
- 先读：`templates/deck-editor/README.md`
- 生成时把 `editor.css`、`toolbar.html`、`runtime.js` 内嵌进最终 HTML。
- 按当前 deck 写 `editSelector`、`pageSelector`、`host.afterRebuild` 等配置。

编辑器底线：

- `E` 进入编辑模式，`Esc` 退出；放映态不出现编辑控件。
- 保存只走 Chrome File System Access 授权写当前 HTML。
- 不使用 localStorage 版编辑器。
- 不依赖本地保存服务、`server.py`、`edit-server.py` 或 `POST /__save`。
- 拒绝授权时可以临时编辑 DOM，但状态栏必须提示不会落盘。

编辑器能力清单和验收清单见 `templates/deck-editor/README.md`。

### 7. 自检

交付前至少检查：

- 主线、知识点清单、页序大纲、草模走查（用户明确说锁）、风格预览五道确认都拿到了用户认可；无对话通道时，保守假设已集中列入待确认项。
- 主线是一条追问链，每一页能回答上一页冒出的问题，不是并列清单，也没有主题相关但接不上链条的旁支页。
- 页面顺序是认知顺序，不是制作顺序。
- 定位全程被当尺子反复使用，不是讲一次就丢开；定位陈述扛得住「产品还是作品」的追问。
- 技术可行性验证挂在已有功能设想之后，不是一段独立插入的技术课。
- 每个知识点都在某页明确立起来。
- 开场在第一个知识点前建立了目标感；带做环节有验收位，纯认知内容没有逐点设检验。
- 可迁移方法和案例事实分层清楚。
- 没有编造项目私有数字、故事、坑和产出。
- 左右翻页、目录、进度条、页码跳转可用。工程侧验证按 `references/deck-verify.md` 执行（可直接跑 `scripts/verify-deck.mjs`）：带 cache-buster 全新加载断言启动成功、逐页版面量测、资源引用清单、slide 与 `SECTIONS` 一致性。
- 排版验证在固定舞台的设计尺寸下做，并换 2 到 3 个差别大的窗口尺寸和比例（含非 16:9）确认每页渲染完全一致。这是在验证舞台真的固定、内容没偷用视口相对单位：换个尺寸排版就变，说明没做固定舞台，回去补，不许拿某一个预览尺寸的结果当排版结论。
- 页面长度超出实际截图能覆盖的范围（如内嵌的长回放页）时，改用 DOM 查询、hash 跳转等断言核验，不依赖截图。
- 编辑保存功能可用，拒绝授权时有明确状态提示。
- 中文断行、字号、对比度、reduced-motion 没有明显问题。
- 每页有足够的素材、图形、演示或结构化排版，不是纯文字堆叠。
- 每页首屏非空：主视觉立即可见，frag 只用于次要 remark。
- 主要页型已经形成体系：封面、Q页、截图页、技术动画页、步骤轨、总结/作业页各自有稳定视觉规则，不是逐页临时美化。
- 截图框按素材类型统一适配，网页、移动端、终端、文档/报告、数据看板等外壳不混用；同类截图没有裸图和多套阴影圆角并存。
- 章节轨、步骤轨、面包屑等导航提示已弱化，当前页主体比进度提示更突出。
- 技术机制动画统一放进演示舞台，节点、箭头、标签、提示气泡的视觉语言一致，并按主题主色做适配。
- Q页或章节转折页形成统一追问系统，能看出「当前问题」和「下一段回答」，不是零散的大字页。
- 文案符合 no_ai_style：不自我标榜、不解释用意、不反问式收尾、不表态铺垫、不煽情、不空附和，禁用破折号（——）和弯双引号（“”），中文与数字或英文之间不加空格，中文语境不混英文标点。

工程验证优先用 skill 自带的 `scripts/verify-deck.mjs`；确需临时脚本，用完即删。项目内的非平凡改动要追加 `WORKLOG.md`。

## 修改已有 deck

已有 deck 的页之间常有隐性耦合。动结构前先找这些地方：

- `SECTIONS`、目录、页码、进度条；旧 deck 还可能带 `NOTES` 讲师注记（已取消的历史模块，新 deck 不再内置）
- 按页码触发的动画、demo、计数器或特殊效果
- hash 跳转、键盘翻页、触摸翻页
- 编辑器重建页面后的同步逻辑

改动分流：

- 插删页 ≥ 3、动章节结构、或改追问链走向的，先出 mini 页序大纲（只列受影响区段）给用户确认，落盘 `docs/lesson-<YYYYMMDD>-<slug>-plan.md`；同一份 deck 的历次大改共用一份 plan 文档，追加「修订记录」，不新开文件。
- 其中涉及页序重排或改链走向的，mini 大纲确认后不直接动正式文件：先复制一份 deck，用 `scripts/deck-surgery.mjs` 在副本上重排出结构预览版，用户在副本里走查认可，再对正式文件重放同一串操作落定；定序理由记进 plan 文档，之后要再翻先对质理由。只插删页、不动顺序的，确认后直接改正式文件。
- 纯文案、版式、单页微调，直接改。
- 视觉升级（换风格、统一页型、整批收口）先在 1 到 2 个代表页打样给用户看，认可后再铺全 deck，不许直接铺。

编辑态与版本安全：

- 动手前确认用户浏览器里的编辑态已关闭或已刷新：编辑器保存走整文件序列化，旧编辑态一次保存会把 session 刚写进磁盘的改动整份冲掉；改完提醒用户刷新页面再继续编辑。
- 大改前先备份一份快照到 scratchpad。
- 收到按页码的批量反馈，先核对磁盘版本和用户所见版本是不是同一份（页数、抽查几页标题），不一致先对齐再动手；沟通和文档里页码旁边带稳定 id。

修改规则：

- 换案例只改案例事实，不动通用方法骨架。
- 整页的移动、插入、删除用 `scripts/deck-surgery.mjs` 按 slide id 操作，自动同步 SECTIONS 和存量 NOTES，不手搬 HTML 块；重排后仍要人工检查被移动页的承接文案。
- 插页、删页、重排后，重算所有按页码索引的表。
- 能用稳定标记定位特殊页，就不要写死页序。
- 改完必须浏览器从头翻到尾，查总页数、章节覆盖、特殊行为和控制台。凡动过 `<script>` 里的数据（`SECTIONS`、遗留 `NOTES`、demo 注册表），必须带 cache-buster 全新加载并断言 deck 启动成功，hash 跳转式验证测不到加载期语法错误。方法见 `references/deck-verify.md`。

边界：

- 动手改结构前先看一遍 `git status` / `git diff`，确认没有会被覆盖的未提交改动（含用户自己刚做的手工编辑）。
- 改完做一致性核对：slide 实际数量与 `SECTIONS` 覆盖区间一致且连续。旧 deck 若带 `NOTES`，条目数必须同步（插页补空字符串占位即可，不再写注记内容），索引错位或字符串语法错误会让整个 deck 脚本崩溃。对不上说明有页面在改动中丢失或错位。

## 不做什么

- 不按项目制作时间线编课。
- 不把不同层级的点并成一排。
- 不把通用方法和单个案例糅在一起讲。
- 不把用户输入原样转写成课件。
- 不编造项目里没有的事实。
- 不新增运行时库依赖。
- 不用视口相对单位（vw/vh、clamp 里的 vw）做正文字号和留白。deck 会随窗口重排，某个窗口的挤满或溢出在别的窗口看不到，作者验证时抓不到。舞台内一律用相对固定舞台的单位。
- 不让编辑功能污染放映态。
- 不做讲师注记（NOTES / 注记抽屉），解释性内容不上墙也不另找地方存放。
- 不为了主题相关插入接不上当前追问链的案例或反例，哪怕内容本身没错。
- 不用产品话术包装一个只解决讲述者个人或团队内部问题的自用工具。
- 不把技术可行性验证做成脱离功能设想的独立技术课。
- 不展示作者自己的调试事故、失败版本、多版本打分记录，除非用户特别要求；确有教学价值时改写成学员能用的方法论（如把「踩过的坑」提炼成「常见误区」），不原样呈现事故过程。

## 交付格式

最终回复按这个结构给：

- `lesson.html`：文件路径
- `操作`：翻页、目录、页码跳转、编辑模式入口
- `课程结构`：章节、知识点、页码、预计分钟
- `课前准备`：提示词、素材、备用演示、待用户确认项
- `资源清单`：deck 引用的全部相对路径文件，发布时按清单核对
- `已验证`：浏览器实跑结果、控制台、编辑保存
- `工作日志`：`WORKLOG.md` 追加位置

如果没有做其中某项，明确说没做和原因。

## 失败兜底

- 素材没确认：用占位，写入待确认项。
- 没有真实项目坑或数字：不用假的故事顶替，可以用日常比喻并标明是比喻。
- 载体太重：换轻载体，说明降级，并列入待确认。
- 时长不够：砍知识点，保主线，不把每个点压成走马灯。
- 概念太难：拆成多页，用素材和图形摊开讲。
- 知识清单未确认：按保守假设搭骨架，把假设集中列出来。
- 定位撑不住产品格局，只服务讲述者个人：老实定义成个人作品/工具，不用产品话术硬撑。
