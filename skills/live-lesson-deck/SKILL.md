# 核心原则

1. 产出物：一份网页课件，浏览器打开、左右翻页、像 PPT 一样播放，并且有讲师能对着讲的课程逻辑
2. Skill定位：本skill最重要的工作不是排版，而是把项目或主题整理成学员能学会的认知顺序，**杜绝按项目工作日志顺序整理成课件**
3. 你的定位：你不是deck编排师，你是核心内容创造者，用户给的是项目事实，但一门课还需要机制、原理、术语和迁移方法
   1. 搜索：针对概念和知识点，要主动搜索并验证，是否为最新认知。
   2. 见解：当你对某个知识点和方法论，你有独特的见解时，一定要进行补充，特别是能把学员从「知道怎么做」带到「知道为什么」的内容，要主动补。
   3. 失败：只听用户口述指令、原样转成页面而不做知识增值，是明确的失败模式。
   4. 阶段：走3.5口喷链路时，本条的补充义务全部在口播稿初稿阶段行使；转写稿落定后发现的缺口一律报用户补讲，不自行补写。
4. 分镜思维：你的每页课件不是一页讲5分钟，而是分镜，可能每一页都是1~2个分镜，一个知识点，为了把它讲透，甚至会拆10页及以上，因此请用你的动效、动画能力，全力把一个概念讲透，**不要把一页融入太多概念，也不要课件中写解释性文本**，如果放了解释性文本，就说明动画和动效解释的还不够，要改动效和动画：能用画面承载的解释转成动效，纯展开性的讲解删除、归讲师口播。术语换算：分镜=一个认知任务的画面单元，一页1~2个；拍（步）=分镜内挂翻页键的推进点，一个分镜可含多拍
5. 引导而非平铺直叙：一定要引导学员理解，而不是把一个东西用四宫格、四列、表格形式呈现出来，要思考学员怎么才能理解，为什么会出现这个东西？学员会不会对它有疑问？怎么想到的？追问链是否成立？追问链定义见核心判断模块。方法论知识点要呈现推导现场（打分表逐行出、候选逐个淘汰、当场把方法跑一遍），不出结论卡；此病复发过三次，修复范式是页内动画分步推导，不加页

## 分流

- **新建课件**：从项目、主题、文章、案例或经历生成新 `lesson.html`。先定课程主线，再制作页面，走完整工作流。
- **修改已有deck**：换案例、插页、删页、重排、改文案、加编辑器、修交互。先识别页间耦合，再动结构，走「修改已有deck」章。
- **分流前先查现状**：项目目录里已有 `lesson.html` 时，不管用户措辞像哪类，先向用户核对现状（页数、迭代记录、遗留待办）和本次意图（看现状 / 收尾 / 推翻重做），确认后再选分支，不按字面默认新建。

## 核心判断

1. **困惑链**（即追问链，两名同义）：主线是「学员刚冒出的问题 → 下一段内容回答它」的因果链，不是主题分类。检验每一页：它在回答上一页冒出的什么问题？答不出就删或挪，主题相关也不留。
2. **定位是尺子**：定位贯穿全程反复用（功能取舍、页面去留都拿它解释），不是开场口号页；出现时机可以延后，等学员冒出「这就够了吗」再回收。
3. **定位对象**：先确认某个项目的对象是谁，如果是个人提效工具，那就是用户个人，如果是服务于团队或用户，那就是ToB或ToC产品
4. **新概念**：用户都是运营、数据分析师，如果出现技术型名词或AI时代的全新名词，用学员可能听过的旧概念，通过联想 + 引导，让用户慢慢学会这个概念或知识点。类比物取材按优先级选：此刻课堂现场的事 > 数据分析师的本行概念 > 职场关系 > 日常生活物件，绝不用学员没摸过的东西。类比是一次性消耗品：复讲型课程（同单元多场次）里同一知识点第二遍出现时，裁掉比喻、直接挂术语、压成清单；金句和案例话术是复用资产，逐字保留。

## 工作流

六道门：课程内核（第2步）、知识点清单（第3步）、口喷转写（第3.5步）、页序大纲（第4步）、草模走查（第4.5步）、风格预览（第5步），每道门必须拿到用户明确认可，AI自己判断「已经够清楚」不算数；第3.5步的过门凭据是拿到用户的转写稿，或用户明确说本次跳过；

门状态记账：每过一道门，在项目WORKLOG「当前状态」节记一笔门号和日期（门号用步骤号，如「门2至3.5已过0708，门4至5已过0710」）。后续会话判断某道门走没走过，以这条记录为准，不靠翻历史条目推断。

只有无对话通道（被当subagent一口气执行）时才允许保守假设过门，假设集中写进待确认项。

确认载体：短的发对话；超过约30行的清单或大纲落盘 `docs/lesson-<YYYYMMDD>-<slug>-plan.md`。清单和大纲写同一份，标题和课程摘要后立即放最新修订区；修订记录按时间倒序插入区头，旧记录保留，不追加到文件末尾。对话里发摘要加路径；用户提修改后只重发被改动的部分。

### 1. 收集上下文

向用户收集足够的信息，并落盘到项目的md文档，如果课程定位已经存在，那就无需再问，只需收集其他信息即可

- **学员侧**：学员是谁、基础如何、主动学还是被动刷到、最想学什么、最容易卡在哪。
- **课程定位**：整个课程是教什么的？哪些教，哪些不教，怎么教？
- **章节定位**：从整个课程体系的角度来判断，这一节课在系列里教什么，方法论还是实操？什么讲、不讲什么？
- **项目侧**：读README、开发日志、设计文档、核心代码、成品，找亮点和可复用方法。
- **知识侧**：外部参照查权威来源；自己补的机制、概念、术语先核验（核心原则3）。
- 缺关键答案就问；无对话通道用保守假设继续，记进待确认项。

### 2. 定课程内核

1. 内核确认：这节课的角色和重心、学员最想获得的能力、学员最大的卡点。
2. 内核定义：学员学完这节课的核心收获是什么？不是单个知识点，而是某个系统的认知和收获，不是页面呈现顺序；呈现层决定（如定位延后揭晓）留到第4步。
3. 二次确认：主线压成一句话，学员经过这次学习后，发生了什么样的状态改变？例如：
   1. 心态：从焦虑 --> 安心
   2. 知识：不会xxx --> 会xxx
   3. 身份：离职失业 --> 找到工作上岸


确认后要推翻，就再来一遍，不在页面层小修掩盖主线问题。

### 3. 知识点清单

知识点清单 = 困惑链 + 挂在链环上的知识点 + 知识域完整性判断。困惑链是主体，知识点是挂在链上的附属，顺序不能倒：先有链，再往链上挂点。

1. 先产困惑链：从内核出发，把整节课写成「学员刚冒出的问题 --> 回答 --> 回答里冒出的新问题」的因果链，问题可能会有多个。每一环用学员的话写成真实会问出口的问题，不写讲师视角的章节名。检验每一环：读完上一环的回答，这个问题是不是自然冒出来的？冒不出来就是断链，先修链再挂点。

2. 困惑链举例：LLM不好用 --> 为什么不好用 --> 只会说，不会干活 --> 为什么呢？--> LLM的整个训练过程 --> 它本质就是个预测模型 --> 那我想让他干活怎么办？--> 干活的本质是运行电脑命令 --> Harness在你本地运行命令 --> Harness怎么与LLM交互 --> 还有很多软件、接口不在你本地运行，而是外部软件提供的服务 --> 引入MCP --> MCP是如何跟Harness和LLM交互？--> 理论上LLM + Harness + MCP能完成所有事了，但还是有很多任务它解决不了 --> 人类给它写prompt --> 但prompt太多太复杂，不好管理，而且会占用过大上下文 --> 引出SKILL，SKILL是怎么运作的？ --> 至此，引出这个时代学AI就是要学这四个东西——LLM + Harness + MCP + Skill

3. 知识点逐个挂环：每个知识点标注它回答链上哪一环。挂不上任何一环的，列入待确认项。

4. 知识域完整性判断（防碎片化）：每个知识点先归入所属知识域（如发HTTP请求属于爬虫、建表写入查询属于数据库），再站在域的整体视角过一遍：学员要把这件事真做成，这个域里还缺哪些必学的？哪些明确不教？缺的补进清单标AI补充，不教的写成不教清单一并让用户确认。取舍拿课程定位当尺子（核心判断2），不是把域内知识全塞进来。

5. 每个知识点能支撑课程定位、能落到具体页面；缺的机制原理术语主动补齐，每个补充点按「能独立撑起一页、学员只看这页能答出为什么」给足体量（核心原则3），**但请不要过多补充文字说明，能用素材说明就不用加文字解释**。补充点上页的承载物必须是分步机制图形或具体例子，只有标题加金句的页不合格；图形确实承载不了的抽象点报用户裁决（口播扛或砍点），不用文字解释顶替；一口气模式没有裁决通道时不砍点，先用最简分步图形或占位承载并列入待确认项，砍点永远是用户的决定。

6. 命名：板块名用行业通用词，不自造口语化标签；知识点用一句大白话说清讲什么，不起自造方法名。大白话命名只用于清单确认，不直接作上墙标题。

7. 二次确认：确认格式照 `references/confirm-formats.md` 发（困惑链逐环视图+挂环知识点表+不教清单+待确认项），等用户确认再进3.5口播稿。



### 3.5 口播稿与口喷转写（叙事源）

知识点清单确认后，先出口播稿、等用户口喷，再排页。道理：课的本体是讲述，deck是讲述的视觉辅助，叙事源必须是用户自己的话，不是AI凭项目文件的创作。

1. 加载juzhang-lesson-script skill出口播稿初稿：按困惑链顺序把整节课展开，落盘 `docs/lesson-<YYYYMMDD>-<slug>-script-draft.md`。写成能念的口语，语言层节奏（确认句密度、先考后讲互动位、类比工序、降压收口）照juzhang-lesson-script执行，类比取材按核心判断4的优先级选；一段对应链上一环，AI的知识增值（补机制、原理、术语，核心原则3）全部发生在这一稿里，让用户重讲时一并消化和取舍。juzhang-lesson-script未安装时退回旧口径，由AI直接创作口语稿。
2. 交给用户口喷，同时给两条提醒：一是重讲不是照念加微调，看完一段就把视线离开稿子，用自己的话讲，照念会把AI措辞原样带进课件；二是卡壳的段落不要硬讲，卡壳位置就是困惑链的裂缝，报回来先修链（回到第3步局部修正）再继续。
3. 拿到转写稿后它就是唯一叙事源：第4步排页、第5步上墙字都从转写稿提炼，任务是压缩用户的原话，不是另起炉灶创作。提炼是取内容不是取句式：口语句压上墙时收敛成名词标签或知识名词标题，讲述性原话由讲师口播还原。AI初稿就此作废，禁止拿初稿当叙事源，初稿看着更整齐正是陷阱。转写之后不再新增叙事内容，发现缺口报给用户补讲。
4. 转写稿到手先对照知识点清单出三栏核对表（已覆盖、讲得含糊、完全缺失），含糊和缺失的知识点交用户补讲或明确删除；核对表处理完才进第4步排页。
5. 项目文件从这步起降级为素材源：截图、代码、真实演示从项目取，叙事从转写稿取。
6. 用户明确说本次跳过口喷时，退回项目直出的旧链路（叙事由AI按知识点清单创作），文案自检照常。

### 4. 排教学顺序

1. 核心是接实追问链：排完顺一遍，每页在回答上一页冒出的问题，接不上就挪或删。
2. 引导而非平铺：引导学员，每个知识点先讲为什么需要，再讲它是什么。
3. 提问大呈现：困惑链的主环转折必须独立成问题页（页型照 `references/visual-system.md` 的Q页），问题占大版面，一页只放这一个问题，可配学员视角的具象场景；禁止把主链问题缩成解释页角落的一行小字。全deck节奏是问题页 --> 解释页（一到多页）--> 新问题页的循环，只有次级小疑问允许页内带过。页尾下一问只作衔接预告，主环转折仍须独立Q页；互动收答案的大字提问页是另一页型，不计入Q页体系、不顶替主环Q页。
4. 一环不等于一页：一环用几页由信息量定，常态是问题页加多张解释页，一环拆成一个多页小模块很正常；禁止机械地把每环压成一页。只有一种情况允许单页扛下一环：这页信息量特别大，且全程由一场受控分步动画承载（约五到十步在同一画面上连续推进，如机制链路逐站点亮）。引用本条豁免的页，草模上必须附分步脚本（每步画面变化一句话）；铺内容时动画装不下要拆页的，不算锁后例外重排，直改并报备一句。没有这种动画承载的内容，宁可多拆几页。
5. 素材从认知任务推导：填「页型与素材需求」时，先写这页的认知任务（学员要从不懂什么到懂什么），再按优先级选素材：真实演示或会话回放 > 真实截图和代码 > 对比图、流程图 > 数据图表 > 文字卡。文字卡是兜底不是默认，用了必须配语义图标。用户点名的自产成品素材不参与本条优先级取舍。走了3.5的，素材语境从转写稿取：用户讲这段时举了什么例子就用什么例子；转写稿没有的例子列入缺口报用户，不自造。
6. 一次只引入一个陌生概念，难点之间用低负荷点缓冲；一页只承担一个认知任务（核心原则4）。
7. 目录按学员心智切4到6章，不写「钩子」「铺垫」这类讲师术语。
8. 一组步骤或部件要有总览页，深钻知识的每页，必须用步骤轨回扣全貌。
9. 图里小框标题用名词，2到7个汉字；超长先删成名词，删不动才加宽节点或把说明移出。
10. 外来方法论要搭教学坡道：先讲理论为什么出现，用案例把框架讲透，套用段显式挂框架步骤；通用框架页和本课应用页分两层，禁止从本课答案倒推框架措辞。走3.5链路时，框架层的叙事缺口报用户补讲，不自行创作。
11. 二次确认：页序大纲照 `references/confirm-formats.md` 发（章节分组+每页回答的问题+素材缺口），用户认可后进4.5，不直接制作。

### 4.5 草模审查（结构冻结）

页序确认后先草模再制作：教学判断只在「像上课走流程」时被激活，顺序问题在草模上改是几分钟，在成品deck上是百页级联动。

1. 草模=真实壳+骨架页：固定舞台壳直接搭好（锁定后铺内容不重建），每页只放标题、「这页回答的问题」一行、素材占位块；素坯样式，不代表最终风格。
2. 用户像上课一样从头走查：每翻一页能顺畅说出下一句口播，讲不顺就是链断，当场用 `scripts/deck-surgery.mjs` 调序。
3. 走查阶段反复翻顺序不算返工，这正是用户想清楚教学法的过程。
4. 过门：用户明确说锁；锁定时每个定序决策一句话理由记进plan文档。
5. 锁后再动模块级顺序按例外处理：先对质当时的定序理由，新论点成立才动，防止来回震荡。
6. 锁后进第5步，按模块铺内容、一个模块一轮反馈；此时反馈是页内修改，不再级联重排。

### 5. 制作deck

1. 视觉、舞台、翻页、动效交给 `frontend-slides` skill（4.5搭壳前加载）；跳过它的Phase 0和1（内容侧本skill已定完），风格环节不跳。
2. **固定舞台是硬要求**：固定尺寸舞台+transform:scale等比缩放铺满窗口，舞台内一律用相对舞台的单位，禁vw/vh和clamp(...vw...)。不这样做布局随窗口重排，单尺寸验证抓不到别的尺寸的问题。模板见 `references/fixed-stage.md`。
3. 二次确认：风格必须用户看过实际画面才算确认。没指定风格出3版差异化首屏预览挑；点名了风格或有视觉基线也要实作1版首屏预览请认可，不许因「基线已存在」跳过。一口气模式同样先做3版差异化首屏，自选性格最强的一版铺开，另两版随交付给用户挑；不许省掉比稿直接落一版保守风。
4. UI按页型成体系，不逐页美化：动手做视觉前必读 `references/visual-system.md`（页型清单、风格适配、视觉收口）。
5. 概念视觉身份先于单页变化：连续机制页动手前先列概念分级和视觉词典；同一概念跨页固定颜色、形状、位置与层级。颜色表示身份，形状表示类型，动画只表示状态变化。协议、方法、消息不要画成与角色同级的人物卡。连续页复用同一张总图，只做高亮、遮罩、局部放大和消息移动。概念跨多页时先做2张代表页样板，用户确认后再铺开。
6. 素材：真实产品、代码、配置必须展示真实内容，拿不到就占位并记待确认项；外部产品用真logo（放 `assets/<category>-logo/`，相对路径引用）。「人和AI真实协作过程」优先做真实会话回放，提示词卡片只用于照抄执行场景（回放页型见 `references/page-patterns.md`）。素材双轨：外部长文往课里融走蒸馏（为什么+怎么做两拍）；用户点名要上deck的自产成品（界面图、写好的方案）全量呈现，不按教学判断自行筛删，删减需用户定向授权。用户主动交来并说要放进课里即算点名；全量呈现与页面密度冲突时分页摊开，不压缩、不转述。
7. 载体：单文件 `lesson.html` 浏览器直开；静态素材可随 `assets/` 交付，不依赖本地服务和构建；web字体必须带系统fallback；每个slide给稳定id（页码会漂，定位和沟通都用id）。
8. 代码展示讲行为、结构和原则，不围绕变量名、函数名、类名展开。
9. 图标：清单、对比、文字卡这类文字偏多的页面，主动去开源图标集（lucide、tabler等）搜语义匹配的图标，以内嵌SVG落地，不引运行时依赖（红线9）。图标要贴知识点语义，纯装饰性堆砌不算配齐。

### 6. 内置作者编辑模式

- 新建课件默认内置；模板在 `templates/deck-editor/`，接入方式和验收清单照其README。
- 底线：`E`进入`Esc`退出，放映态零编辑控件；保存只走Chrome File System Access直存当前HTML，不用localStorage、不依赖本地保存服务；拒绝授权可临时编辑但状态栏提示不落盘。

### 7. 自检

1. 六道门都拿到用户明确认可；无对话通道时假设已集中列入待确认项。
2. 红线清单逐条扫，全部不踩。

3. 核心原则5条和核心判断4条逐条对照成立。

4. 工程验证照 `references/deck-verify.md` 跑 `scripts/verify-deck.mjs`（全新加载boot断言、逐页版面、资源清单、SECTIONS一致性）；固定舞台换2到3个差异大的窗口尺寸确认渲染一致；截图覆盖不了的长页面用DOM断言；编辑保存可用；确需临时脚本用完即删。

5. 每页首屏非空、素材图形化不是文字堆叠、页型成体系，对照 `references/visual-system.md` 收口清单过一遍。

6. 文案做两遍扫：一遍按红线「no_ai_style」逐项扫；一遍按teaching-voice上墙形态条款，逐页查标题、眉题、标签、节点名有没有描述句和疑问句（Q页、站队页、五拍页尾下一问除外），有就改成名词。

7. 非平凡改动追加项目 `WORKLOG.md`。

8. 外派审查不默认开，轮次跟用户节奏，用户赶时间时只做本节自检；审查产出的问题清单先由主线对源码核实、剔除误报，再动deck。

9. 逐知识点自问：学员只看对应页面序列（含动画各步）能答出的「为什么」是什么？答不出的知识点列入修改清单再交付。

## 修改已有deck

修改的时候，方案过了确认门就大胆执行新增、删除或调整章节顺序，不必畏手畏脚；动大刀子前提醒用户commit

动结构前先找隐性耦合：`SECTIONS`、目录、页码、进度条；按页码触发的动画、demo、计数器；hash跳转、键盘和触摸翻页；编辑器重建后的同步逻辑；存量`NOTES`（已取消的历史模块，新deck不内置）。

分流：

1. 插删页≥3、动章节结构、改追问链走向：先出mini页序大纲（格式照 `references/confirm-formats.md`）给用户确认，落盘该deck的plan文档。
2. 其中涉及页序重排或改链走向的：确认后先复制副本，用 `scripts/deck-surgery.mjs` 在副本上排出结构预览给用户走查，认可后对正式文件重放同一串操作；定序理由记plan文档，再翻先对质理由。只插删不动顺序的直改正式文件。
3. 纯文案、版式、单页微调：直接改。
4. 视觉升级：先1到2个代表页打样给用户看，认可后再铺全deck，不许直接铺。

版本安全：

1. 动手前确认用户浏览器里的编辑态已关闭或已刷新（编辑器整文件序列化保存，旧编辑态一次保存会冲掉刚写盘的改动）；改完提醒刷新再编辑。
2. 大改前备份快照到scratchpad；动结构前看 `git status` / `git diff`，确认没有会被覆盖的未提交改动（含用户手工编辑）。
3. 收到按页码的批量反馈，先核对磁盘版本和用户所见是不是同一份（页数、抽查标题）；沟通和文档里页码旁带稳定id。

修改规则：

1. 整页移动、插入、删除用 `scripts/deck-surgery.mjs` 按slide id操作（自动同步SECTIONS和存量NOTES），不手搬HTML块；重排后人工检查被移动页的承接文案。
2. 插删重排后重算所有按页码索引的表；特殊页用稳定标记定位，不写死页序。
3. 改完从头翻到尾，并做一致性核对：slide数与SECTIONS覆盖连续一致；存量NOTES条数同步（插页补空字符串，不写内容）。凡动过`<script>`里的数据，必须带cache-buster全新加载断言启动成功（hash跳转测不到加载期语法错误），方法照 `references/deck-verify.md`。

## 红线

内容：

1. 不按项目制作时间线编课（核心原则2）。
2. 不把用户输入原样转写成课件（核心原则3）。走3.5口喷链路时知识增值已前移到口播稿初稿，从转写稿提炼上墙字不算此项违规。
3. 不编造项目私有事实：数字、故事、坑、产出。上墙的价格、耗时、效果类承诺必须有证据支持；拿不到真实数据就占位或只给计算框架，不冒充实账。
4. 不插主题相关但接不上追问链的内容；不把不同层级的点并成一排；不把方法和单个案例糅成一页。
5. 不评估、不预算、不提醒课程时长。时长由用户自己管控，AI对时长的估计不可靠（2026-07-15用户明确）。SECTIONS等时间字段照用户口径填写；审查、走查、交付里不主动提出时长问题，用户问起才答。
6. 不原样展示制作与调试过程的流水账：AI调课件、作者调prompt的事故记录、失败版本、打分记录不上墙（用户点名要除外），有教学价值就提炼成「常见误区」。项目开发中的真实踩坑是教学素材，提炼出症状、关键动作、结论后可用破案叙事等页型讲。
7. 不做讲师注记（NOTES/注记抽屉）；解释性、展开性内容不上墙也不另找地方存放。
8. **no_ai_style**：路径 `~/.claude/rules/no_ai_style.md`

工程：

9. 不新增运行时库依赖。
10. 不用视口相对单位（vw/vh、clamp里的vw）做内容尺寸，舞台内一律用相对舞台的单位。
11. 不让编辑功能污染放映态。

## 交付格式

最终回复按这个结构给，没做的项明确说没做和原因：

- `lesson.html`：文件路径
- `操作`：翻页、目录、页码跳转、编辑模式入口
- `课程结构`：章节、知识点、页码；时长只在用户给了口径时附（红线5）
- `课前准备`：提示词、素材、备用演示、待用户确认项
- `资源清单`：deck引用的全部相对路径文件，发布时按清单核对
- `已验证`：浏览器实跑结果、控制台、编辑保存
- `工作日志`：`WORKLOG.md` 追加位置

## 失败兜底

- 素材没确认：用占位，写入待确认项。
- 没有真实项目坑或数字：不用假故事顶替，可以用日常比喻并标明是比喻。
- 载体太重：换轻载体，说明降级，列入待确认。
- 用户反馈时长不够：砍知识点保主线，不把每个点压成走马灯。
- 概念太难：拆成多页，用素材和图形摊开讲（核心原则4）。
- 知识清单未确认：按保守假设搭骨架，假设集中列出。
- 定位撑不住产品格局：老实定义成个人作品/工具，不用产品话术硬撑。
