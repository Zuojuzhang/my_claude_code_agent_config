# AI/Web3 长文写作风格指引

## 术语规范

### AI 领域

| 推荐用法 | 避免用法 | 说明 |
|---------|---------|------|
| 大语言模型 / LLM | 大模型 | "大模型"含义模糊，LLM 更精准 |
| 智能体 / Agent | AI 代理 | "智能体"是国内技术圈主流译法，Agent 也可直接用 |
| 提示词 / Prompt | 提示语 | "提示词"更通用 |
| RAG | 检索增强生成 | 首次出现可用"检索增强生成（RAG）"，后续直接用 RAG |
| 微调 / Fine-tuning | 精调 | "微调"更主流 |
| 推理 / Inference | 推断 | 技术语境下"推理"更常用 |
| Token | 词元 | 直接用 Token，中文技术圈已接受 |
| 多模态 / Multimodal | 跨模态 | "多模态"更标准 |
| 向量数据库 / Vector DB | 向量存储 | "向量数据库"更准确 |
| Function Calling | 函数调用 | 首次出现用英文，后续可混用 |

### Web3 领域

| 推荐用法 | 避免用法 | 说明 |
|---------|---------|------|
| 去中心化 / Decentralized | 分散式 | "去中心化"是行业标准译法 |
| 智能合约 / Smart Contract | 智慧合约 | "智能合约"是通用译法 |
| 共识机制 / Consensus Mechanism | 共识算法 | 机制比算法更准确（包含经济激励等非算法部分） |
| DePIN | 去中心化物理基础设施网络 | 首次出现用全称+缩写，后续直接用 DePIN |
| TVL | 总锁仓量 | 首次出现用"总锁仓量（TVL）" |
| DAO | 去中心化自治组织 | 首次出现用全称 |
| ZK / 零知识证明 | 零知识 | "零知识"含义不完整 |
| Layer 2 / L2 | 二层 | 直接用 Layer 2 或 L2 |
| DApp | 去中心化应用 | 首次出现用"去中心化应用（DApp）" |
| Gas 费 | 矿工费 | "Gas 费"更通用且跨链兼容 |

### 跨领域术语

- 中英文混用是正常现象，不需要刻意全中文化或全英文化
- 缩写词首次出现时给出全称（中文或英文均可）
- 同一篇文章中，同一概念始终用同一译名，不要混用

## 论述结构建议

### 技术博客

技术博客的核心价值是"帮助读者理解或做到某件事"。结构应服务于这个目标：

1. **问题先行**：先说清楚要解决什么问题，读者为什么需要关心
2. **原理解析**：深入但不过度，重点讲"为什么这样设计"而非"代码怎么写"
3. **实操落地**：提供可操作的步骤或可参考的架构，而非纯理论
4. **踩坑与经验**：真实的使用经验和注意事项比完美方案更有价值

写作中常见的结构问题：
- ❌ 堆砌概念但没有串联逻辑 → 读者记了一堆名词但没理解关系
- ❌ 直接给方案但不解释问题 → 读者不知道这个方案好在哪里
- ✅ 问题→分析→方案→验证，形成闭环

### 行业分析

行业分析的核心价值是"提供独到判断，帮助读者决策"。结构应体现推理过程：

1. **现象描述**：用数据和事实说话，不凭感觉
2. **归因分析**：现象背后的驱动力是什么（技术、市场、政策、人性）
3. **趋势推演**：基于归因逻辑推导未来走向，明确标注确定性程度
4. **行动建议**：不同角色（创业者/投资人/开发者）应该做什么

写作中常见的结构问题：
- ❌ 罗列项目但缺乏比较框架 → 变成项目简介合集而非分析
- ❌ 只有判断没有推理 → 读者无法评估结论的可靠性
- ❌ 预测不标注确定性 → 把猜测包装成定论
- ✅ 判断+推理+证据，让读者可以自行评估

## 写作风格

### 语调

- 专业但不学究：像跟同行聊天，不像在写论文
- 有立场但不偏激：行业分析要有判断，但标注推理过程让读者自行验证
- 直接但不粗暴：观点清晰表达，不用"可能"、"也许"削弱力度，但也不用绝对化表述

### 信息密度

- 每个段落必须有信息增量。如果删掉某段不影响理解，说明这段不该存在
- 避免用不同的话反复说同一件事（除非是核心论点的刻意强调）
- 数据和案例是信息密度的来源——用具体数字替代模糊描述

### 技术深度

- 技术博客：假设读者有基础技术背景，不需要解释"什么是 API"这种级别的基础概念，但要解释该领域的专有概念
- 行业分析：技术部分点到为止，重点在于技术选择带来的业务和市场影响
- 不确定读者背景时，偏技术侧的文章宁可深一点——浅了没有价值，深了读者可以跳过

## 常见写作问题

### 注水信号

出现以下情况说明内容在注水，需要精简：
- 一段话可以压缩成一句而不丢失信息
- 连续多个段落都在用不同方式说同一件事
- 大段引用但没有自己的分析和点评
- 用形容词替代论据（"极具潜力"→ 具体是什么潜力？数据呢？）

### 逻辑断裂信号

- 段落之间缺乏过渡，读起来像拼凑
- 前文提出的概念后文突然换了个名字
- 引言中承诺的内容正文没有覆盖
- 结论与前文论据不匹配

### 术语混乱信号

- 同一概念出现多种叫法（如"智能体"和"AI Agent"和"代理"混用）
- 缩写未定义就使用
- 中文译名与行业通行译法不一致
