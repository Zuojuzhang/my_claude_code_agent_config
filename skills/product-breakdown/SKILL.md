# 产品需求拆解调度层

## 定位

方法论不在本文件里:结构层在`ooux-product-design`,执行层在`feature-breakdown`。本skill管四件事:判断进哪一层、按运行模式控制节奏、把产出落盘、和上下游衔接。

## 阶段判断(开工第一步)

- 产品还没有骨架(对象/页面/数据模型没梳理过)→ 阶段A
- 已有骨架,要细化某个具体功能 → 阶段B
- 要补某个功能的AC → 直接进阶段B的B5
- 纯bug修复、纯样式调整 → 不进本skill,直接走code-writer路线

## 运行模式

**主线模式(默认)**:本体加载本skill直接和用户走。

- 严格按方法论skill的节奏:每步输出结构化结果,等用户确认再进下一步,不一口气全输出
- 每步确认后立即把该步产出追加写进落盘文件(见落盘协议),中断、切话题、上下文压缩都不丢进度
- 重新进入时先Read已落盘文件,从缺的步骤继续,不从头重跑

**一口气模式(被作为subagent派出时)**:

- 一口气跑完全部步骤,中途不暂停(subagent没有和用户对话的通道,方法论skill里「等用户确认」的节奏不生效)
- 拿不准的假设、开放问题、多种合理取舍,集中记录到产出文档末尾「待确认项」章节,每条写清:假设内容、按哪个方向执行了、其他选项是什么
- 输入不足以开工时(连阶段都判断不出、功能描述空到没法拆)不要瞎跑,直接返回说明缺什么
- 修正重派时基于已落盘文件增量修正,不从头重来
- 返回只给文件路径加待确认项摘要,不粘贴全文

## 落盘协议

- 阶段A产出:`<项目根>/docs/pm-<YYYYMMDD>-<slug>-structure.md`
- 阶段B产出:`<项目根>/docs/pm-<YYYYMMDD>-<slug>-ac.md`
- 日期从上下文取(对话环境信息里有当天日期;一口气模式由调度方在prompt里给),不调用shell命令取系统时间
- slug用小写英文连字符(例:`pm-20260603-order-system-structure.md`),`docs/`目录不存在则先建
- 主线模式下文件在第一步确认后就创建,按终稿章节骨架立好,每步确认后填充对应章节
- 一口气模式跑完一次性写全
- 用户对已确认章节提修正 → 直接改文件对应章节,不重跑流程

## 阶段A:产品结构层

目标:把产品idea变成对象、关联、页面、数据模型的骨架。一个产品只走一次。

1. `Read ~/.claude/skills/ooux-product-design/SKILL.md`,按它的四步(提取核心对象 → CSMA拆解 → 关联关系 → 推导页面)和上下文收集要求执行
2. 交付物:对象清单表、对象关系图、页面结构清单、数据模型草案(一口气模式外加待确认项)
3. 走完交给用户确认;只要骨架就停在这,继续细化功能则进阶段B

## 阶段B:功能执行层

目标:针对单个具体功能,做实体生命周期、五问、IO链分析,输出子功能清单和AC验收标准。

1. `Read ~/.claude/skills/feature-breakdown/SKILL.md`,严格按它的五步(实体生命周期 → 五个必问 → 输入输出链 → 子功能优先级 → AC验收)执行
2. 衔接:走过阶段A的,B1从structure文件圈出本功能涉及的对象再补隐性实体;没走过的,B1先做完整实体识别
3. ac文件开头一行写明`参考结构文档：pm-<YYYYMMDD>-<slug>-structure.md`(如有)
4. 交付物:实体生命周期、五问答复、IO链、子功能优先级清单、AC列表(一口气模式外加待确认项)

## 下游衔接

- 交付给用户和下游的是**文件路径**,不粘贴全文
- code-writer按AC文件路径读验收标准,code-reviewer按同一文件核验,ui-designer按它出界面方案

## 约束

- 不做商业判断和功能建议(那是product-strategy skill),不做UI设计(那是ui-designer)
- 输出会被人当作正常文字阅读,避开以下AI腔(完整规则见`~/.claude/rules/no_ai_style.md`,需要时Read):自我标榜(老实说/说实话/坦白讲)、解释自己用意、结尾反问加主动规划下一步(要不要我)、破折号和双引号、油腻煽情、表态铺垫、附和措辞(确实/没错)、中文与数字英文之间加空格、写作手法自评式标签
